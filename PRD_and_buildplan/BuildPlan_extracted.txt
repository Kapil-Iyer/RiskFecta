Phase 2 — Feature Engineering
Sessions 3-4  |  ~8 hrs  |  File: pipeline/features.py

[Cursor Pro]  Build pipeline/features.py

Compute all technical indicators using pandas-ta library (clean, vectorized, backward-looking by default):
RSI: pandas_ta.rsi(close, length=14) — uses t-13 through t only

Bollinger Bands: pandas_ta.bbands(close, length=20)

Phase 3 — LSTM Model (20% Time Buffer)
Sessions 5-7  |  ~18 hrs  |  File: models/lstm.py  |  Highest-risk phase — take it slow

Architecture
[Cursor Pro]  Build models/lstm.py

PyTorch LSTM class: 2-layer LSTM, hidden_size=64, Dropout(0.2), Linear output layer → scalar return prediction
Input tensor shape per sample: (batch_size, 60, n_features) — 60-day sequence, n_features = OHLCV + RSI + MACD + BB + vol + momentum
Target: 30-day forward return (scalar) — normalized with StandardScaler, inverse-transform after prediction
Loss: MSE on normalized returns. Optimizer: Adam, lr=1e-3. Epochs: 50 with early stopping (patience=10)

Rolling Window Training Loop
[Claude Pro]  Review every iteration of rolling-window loop for leakage

Implement exact pseudocode from PRD v4 Section 4 — no deviation
Sequence construction: within each training window [t-252:t], build all valid 60-day sequences ending before t
Valid sequences: start at t-252, end at t-60 minimum — gives (252-60) = 192 sequences per ticker per window
Training data: DataLoader with batch_size=32, shuffle=True within training window only
Prediction at t: single forward pass on sequence [t-60:t] — strictly within or at training window boundary
Refit model from scratch each window step — no weight transfer between windows (prevents temporal leakage via model state)
Store predictions: INSERT INTO predictions (ticker, forecast_date, target_date, lstm_pred) with ON CONFLICT DO NOTHING

⚠ CRITICAL: Paste your rolling-window training loop to Claude Pro BEFORE running it. Leakage here invalidates the entire backtest. The most common mistake: accidentally including sequences that cross the t boundary.

Compute Resources
Train on Google Colab (free GPU) if laptop CPU is too slow — LSTM training on CPU can take hours
Save trained model weights to models/lstm_weights_{window_id}.pt after each window
Inference runs on CPU fine — only training needs GPU

✓ TEST: LSTM predictions populated in predictions table for all rolling windows. Directional accuracy logged per ticker per window. Fallback: if training too slow, simplify to single-layer LSTM with hidden_size=32.

Phase 5 — Ensemble + Evaluation
Session 10  |  ~6 hrs  |  File: models/ensemble.py

[Cursor Pro]  Build models/ensemble.py

Compute: ensemble_pred = 0.5 * lstm_pred + 0.5 * xgb_pred — for all rows where both predictions exist
UPDATE predictions SET ensemble_pred = (lstm_pred + xgb_pred) / 2.0 WHERE lstm_pred IS NOT NULL AND xgb_pred IS NOT NULL
Fill actual_return: when target_date has passed, compute actual 30-day return from prices_raw and UPDATE predictions
Compute directional_correct: sign(ensemble_pred) == sign(actual_return). Exclude rows where actual_return = 0.
Evaluation metrics to log to console and store in risk_metrics table:
Overall directional accuracy: sum(directional_correct) / count(non-zero actual_return)
Per-asset directional accuracy breakdown
R² and RMSE for ensemble vs actual returns
LSTM vs XGBoost individual directional accuracy comparison

Phase 6 — Portfolio Optimizer
Sessions 11-12  |  ~10 hrs  |  File: optimizer/portfolio.py

[Cursor Pro]  Build optimizer/portfolio.py

Inputs: ensemble_pred vector (expected returns per asset) + covariance matrix from actual realized returns in training window
Covariance: cov_matrix = returns_df[t-252:t].cov() * 252 — annualized. Uses actual returns only, not predicted.
SciPy optimizer: minimize portfolio variance subject to target return constraint
Constraints: weights sum to 1, all weights >= 0 (long-only), optional max 20% per asset
Solver: SLSQP — handles linear equality and inequality constraints
Verify covariance matrix is positive definite before passing to optimizer — add explicit check
Generate Efficient Frontier: run optimizer for 50 target return values between min and max feasible return
Compute risk metrics per optimal portfolio:
Portfolio return, portfolio volatility (annualized)
Sharpe ratio: (portfolio_return - USGG10YR/252*252) / portfolio_volatility
VaR 95%: portfolio_return - 1.645 * portfolio_volatility (parametric)
Max drawdown: computed from portfolio return series in prices_raw window
Store all results in portfolios table and risk_metrics table with run_id

[Claude Pro]  Review optimizer math

Paste portfolio.py to Claude Pro — verify covariance matrix uses actual returns, Sharpe uses USGG10YR/252, positive-definite check is correct

⚠ CRITICAL: If optimizer fails to converge: tighten weight bounds (0.02 to 0.15 per asset), check covariance matrix condition number, reduce universe size temporarily for debugging.

✓ TEST: portfolios table populated with Efficient Frontier weights for multiple target returns. risk_metrics table has Sharpe, VaR, max drawdown per run_id. Sharpe ratio is positive for at least some portfolio points.
