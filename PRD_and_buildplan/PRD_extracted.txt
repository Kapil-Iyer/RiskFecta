1. Project Identity
What RiskFecta Is
A quantitative portfolio analytics platform combining ML return forecasting with mathematical portfolio optimization
A hybrid forecasting engine: LSTM (deep learning on price sequences) + XGBoost (tabular macro/factor model) → weighted ensemble
A portfolio optimizer using Modern Portfolio Theory: ensemble expected returns + historical covariance → SciPy Efficient Frontier
An institutional-grade data pipeline: Bloomberg Terminal BQL/Excel → CSV → PostgreSQL → Python modeling
A deployed web application on Streamlit Cloud with interactive Plotly charts + a separate Tableau static report

What RiskFecta Is NOT
Not a trading system, live order execution platform, or paper trading engine
Not a SQLite or PowerBI project — those belong exclusively to PlainCents
Not a Bloomberg API scripting project — data sourced via BQL/Excel export only, terminal-bound
Not a single-model project — the ensemble of LSTM + XGBoost is the core differentiator
Not a regime classifier — XGBoost acts as alternative return predictor, not regime switcher

2. Problem Statement & Solution
Problem
Professional portfolio optimization tools (Bloomberg PORT, FactSet) cost $20,000+ per year and are inaccessible to retail investors. Existing free tools use only historical averages for expected returns, ignoring forward-looking ML signals. There is no accessible platform that combines deep learning return forecasting, classical ML factor models, and mathematically rigorous portfolio optimization in a single deployable system.

Solution
RiskFecta is a Python ML platform that ingests Bloomberg Terminal data via BQL/Excel export, trains a hybrid LSTM + XGBoost ensemble to predict 30-day asset returns, feeds ensemble predictions into SciPy-based Modern Portfolio Theory optimization to compute the Efficient Frontier, and presents results through a Streamlit Cloud web application with interactive Plotly visualizations and a separate Tableau professional report.

4. ML Architecture (Locked)
Overview: Hybrid Ensemble → Portfolio Optimization
The ML pipeline has two parallel prediction tracks that produce a single expected return vector, which feeds into the optimizer.

Track 1 — LSTM (Sequential Price Model)
Framework: PyTorch — custom LSTM class with configurable hidden layers
Input: 60-day rolling window of daily OHLCV data + technical indicators per asset
Technical indicators: RSI (14-day), MACD (12/26/9), Bollinger Bands (20-day), rolling 20-day volatility
Output: Predicted 30-day forward return per asset (scalar per ticker)
Architecture: 2-layer LSTM → Dropout(0.2) → Linear output layer
Loss function: MSE on normalized returns
Training: Mini-batch gradient descent, Adam optimizer, learning rate 1e-3

Track 2 — XGBoost (Tabular Factor Model)
Framework: xgboost Python library — XGBRegressor
Input: Tabular feature vector per asset per month — macro + factor features
Features: VIX level, sector momentum (30-day), market cap (log), beta, dividend yield, 3-month price momentum, 6-month price momentum
All features sourced from Bloomberg BQL export — stored in PostgreSQL features table
Output: Predicted 30-day forward return per asset (scalar per ticker)
Hyperparameters: n_estimators=300, max_depth=6, learning_rate=0.05, subsample=0.8
Tune with GridSearchCV if out-of-sample R² < 0.1

Ensemble — Weighted Average
Final expected return per asset = 0.5 × LSTM_prediction + 0.5 × XGBoost_prediction
Equal weighting as starting point — defensible and simple. Can tune weights in post if time allows.
Ensemble output is a vector of expected returns, one per asset in universe
This vector feeds directly into the portfolio optimizer as the expected returns input

Validation — Rolling Window (Pseudocode Locked)
Method: Rolling window with FIXED train size — NOT expanding window (PlainCents uses expanding; this is an intentional architectural difference)
Train window: 252 trading days (1 calendar year)
Step size: 21 trading days (1 calendar month)
True out-of-sample holdout: final 3 months of data reserved, never touched until final evaluation

Exact pseudocode — implement this loop precisely in forecast.py:
for t in range(252, N - 30, 21):
    train_data  = data[t - 252 : t]      # fixed 252-day window, grows by step
    forecast_date = dates[t]             # prediction made at end of training window
    target_date   = dates[t + 30]        # strictly 30 trading days ahead

    # Fit models on train_data only
    lstm_model.fit(train_data)
    xgb_model.fit(train_data)

    # Predict forward return — no future data visible
    lstm_pred    = lstm_model.predict(data[t - 60 : t])   # 60-day input sequence
    xgb_pred     = xgb_model.predict(features_at_t)       # features at t only
    ensemble     = 0.5 * lstm_pred + 0.5 * xgb_pred

    # Store: actual_return filled post-facto when target_date passes
    insert_prediction(ticker, forecast_date, target_date, lstm_pred, xgb_pred, ensemble)

Look-ahead bias clarification: All technical indicators (RSI, MACD, Bollinger Bands, rolling volatility, momentum) are computed using backward-looking rolling windows only. RSI at date t uses prices t-13 through t. No forward data is used in feature construction. During rolling-window validation, the model sees only training slice features — never features computed from the test window.
LSTM sequence clarification: LSTM training dataset consists of sliding 60-day sequences constructed within the 252-day training window only. lstm_model.fit(train_data) refers to fitting on these sequences — not raw rows. Each sequence is a (60, n_features) tensor. No sequence crosses the training window boundary.

Portfolio Optimization — Modern Portfolio Theory
Inputs: Expected returns vector (from ensemble) + historical covariance matrix (252-day rolling window of actual realized returns — not predicted returns. Covariance computed from actual returns within training window only to prevent look-ahead contamination)
Method: scipy.optimize.minimize with SLSQP solver — minimize portfolio variance subject to target return constraint
Constraints: weights sum to 1, all weights >= 0 (long-only), optional sector weight caps
Output: Efficient Frontier curve + optimal weights per asset for multiple risk targets
Risk metrics computed: Portfolio return, portfolio volatility (annualized), Sharpe ratio (risk-free = USGG10YR / 252 daily proxy — consistent across optimizer, risk_metrics table, Streamlit display, and Tableau report), max drawdown, VaR (95%)

5. Data Pipeline — Bloomberg Terminal Workflow
Architecture Decision: Minimalist Pull + Local Engineering
Bloomberg provides raw atomic fields only. All technical indicators (RSI, MACD, Bollinger Bands, rolling volatility, momentum) are computed locally in Python from raw prices. This keeps the pipeline reproducible, transparent, quota-efficient, and interview-defensible.

In interviews: 'Sourced historical price and factor data via Bloomberg Terminal BQL. Engineered technical indicators locally in Python for reproducibility and model transparency.' This signals both institutional data access and engineering rigour.

Locked Bloomberg Pull — Final Field List
This is the complete Bloomberg pull. Nothing else. Everything else is Python.

Category
Bloomberg Fields + Purpose

PX_LAST, PX_OPEN, PX_HIGH, PX_LOW, PX_VOLUME — raw OHLCV for LSTM input sequences and return calculation

What Is NOT Pulled from Bloomberg

Rolling volatility, momentum_3m, momentum_6m — computed in Python from price history
Implied volatility, breakeven inflation, DXY — not needed for return prediction architecture
PE ratio, EV/EBITDA — fundamental valuation metrics not in locked feature set
30-day realized vol from Bloomberg — computed locally instead

Asset Universe
40-50 stocks: S&P 500 constituents, Technology + Financial Services sectors
Rationale: manageable size for campus terminal quota, sector diversity, strong Bloomberg data availability
Pull all tickers in a single BQL query — most quota-efficient approach

BQL Campus Workflow (Terminal-Bound)
Step 1: On campus terminal — open BQL editor or Excel with Bloomberg add-in
Step 2: Use BQL for multi-asset price panel pull (PX_LAST, OHLCV, TR) — single query for all tickers
Step 3: Separate BQL/BDP query for static fields (BETA, MKT_CAP, SECTOR, DVD_YLD) — one-time pull
Step 4: Pull macro indices (VIX, USGG10YR, SPX) as separate daily series
Step 5: Export all results to CSV files, copy to laptop via USB/email
Step 6: Load CSVs into local PostgreSQL via Python ingest script — all modeling from here
Step 7: Run Python feature engineering pipeline to compute all technical indicators from raw prices

Risk-free rate for Sharpe ratio: use USGG10YR / 252 as daily risk-free rate (already in macro pull). No separate T-bill field needed.

12. Risks & Mitigations
Risk
Mitigation
LSTM training too slow on laptop CPU
Use Google Colab (free GPU) for training. Save model weights (.pt file) locally. Inference runs on CPU fine.
LSTM directional accuracy below 55%
This is expected — financial return prediction is hard. Document honestly. Focus on Sharpe ratio improvement vs benchmark, not raw prediction accuracy.
XGBoost R² very low
Expected for return prediction. Use directional accuracy as primary metric. Tune with GridSearchCV if needed.
Rolling window leakage
Paste loop to Claude Pro for explicit leakage audit before training begins. Do not assume Cursor's code is leak-free.
Bloomberg data pull hits quota
Use BQL (more quota-efficient than BDH/BDP). Do one large pull per data type, not incremental calls. Save all CSVs locally after pull.
PostgreSQL connection on Streamlit Cloud
Use st.secrets for DB credentials. Deployment DB provider: Supabase (default) or Neon — both offer free hosted PostgreSQL, same schema, one-line connection string change. Do not hardcode vendor in code.
Tableau Public file too large
Export individual dashboard views as PDFs, commit to repo/docs folder. Link from README.
Scope overrun on LSTM (Phase 3)
Fallback: simplify LSTM to single-layer, reduce sequence length to 30 days. Ship working model over architecturally perfect one.
Portfolio optimization fails to converge
Add explicit bounds (each weight 0-0.20 max), check covariance matrix is positive definite before passing to optimizer.
